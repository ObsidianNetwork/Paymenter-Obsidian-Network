**Date detected**: 2026-04-24
**Detected by**: User (spotted that PR #11 was merged without the review loop)
**Severity**: Medium (process violation; no production impact observed)
**Status**: Forward-hardening complete; retroactive review of the affected PRs remains infeasible (PR author is immutable on GitHub).

## Summary

Two consecutive PRs (dp-13 #10, dp-12 #11) were squash-merged within 1–3 minutes of opening, with zero CodeRabbit reviews ever submitted. The orchestrator (atlas) accepted the subagents' "merged" claims at face value without verifying that CodeRabbit had actually reviewed.

Root cause turned out to be **the wrong GitHub account ran `gh pr create`**. The host has two authenticated `gh` accounts: `Jordanmuss99` (Pro-entitled CodeRabbit user, subscribing identity) and `ImStillBlue` (a different account, Free tier from CodeRabbit's perspective). The host default active account was `ImStillBlue`, so the subagents' `gh pr create` calls opened those PRs under `ImStillBlue`. CodeRabbit evaluates Pro entitlement against the PR's recorded author and saw a Free-tier user → declined to review.

## Retroactive review attempts (both failed)

1. Posted `@coderabbitai review` on PR #10 at 2026-04-24T04:12:38Z. The comment was authored by `ImStillBlue` (default active `gh` account).
2. Swapped active account to `Jordanmuss99` with `gh auth switch`. Re-posted on PR #11 at 2026-04-24T04:16:12Z as `Jordanmuss99`.
3. Both mentions received the same CodeRabbit auto-reply within ~5 seconds:
   > "The author of this PR is on the CodeRabbit Free Plan. In order to use the Chat feature, please upgrade the PR author to CodeRabbit Pro."
4. Important: "the author of this PR" refers to the PR's recorded GitHub author (`ImStillBlue`), not the comment author. Since PR author is immutable on GitHub, no retroactive path unlocks Pro chat on these PRs.

Implication: A1 (retroactive CodeRabbit review on PRs #10 and #11) is definitively blocked. The only way to get CodeRabbit to review this code would be to close the merges and re-open equivalent PRs — but the code is already in `dynamic-slider`, so a fresh PR would have an empty diff.

## Root causes (ranked)

1. **`gh` default active account was not the Pro-subscribing user.** `ImStillBlue` was the active account on host when subagents ran `gh pr create`. No pre-flight check verified this. This is the primary failure — identity mis-attribution is what caused CodeRabbit to skip review.

2. **No mechanical pre-merge gate.** The `/ralph-loop` contract told agents to "wait for CodeRabbit" in prose. Subagents could claim they had waited without any verifiable artifact. The atlas orchestrator trusted the textual claim instead of querying the PR state.

3. **Timing compounded the failure.** Even if the PR author had been correct, merging within 1–3 minutes of opening is shorter than CodeRabbit's typical 3–8 min auto-review latency. The subagents would have merged before a review arrived regardless. This is a separate timing bug that the hardened contract also closes (rule #2: latest review must post-date last commit).

4. **Stale `@coderabbitai` chat expectations in the old contract.** Prior plans assumed Pro chat was available. When PR-author was wrong (Free-tier by CodeRabbit's view), chat was also blocked, not just auto-review. Future plans must not assume chat as the primary loop driver even on Pro, because the push-driven auto-review works for all entitled tiers.

## Remediation — shipped 2026-04-24

1. **`.sisyphus/templates/ralph-loop-contract.md`** — rewritten. PR author identity section now emphasises `gh auth switch -u Jordanmuss99` + `gh api /user` verification BEFORE `gh pr create`. Commit author email is listed as secondary.
2. **`.sisyphus/templates/ralph-loop-verify.sh`** (new, executable) — enforces five hard rules mechanically: CodeRabbit review exists, review post-dates last commit, zero unresolved threads, `mergeStateStatus=CLEAN`, verdict contains `Actionable comments posted: 0`. Non-zero exit = do not merge.

Amended step 3 of "Out-of-scope finding handling" to use plain GitHub thread comments since `@coderabbitai` chat only works when PR author is Pro.
5. **Git config** — updated both `/var/www/paymenter/extensions/Others/DynamicPterodactyl/.git/config` and `/var/www/paymenter/.git/config` to use `Jordanmuss99 <jordanmuss@hotmail.com>` for future commits.

## Lessons for future dp-NN plans and orchestrators

1. **Identity pre-flight before any PR-creating subagent.** `gh auth switch -u Jordanmuss99 && gh api /user` is non-negotiable. Multi-account `gh` hosts are a foot-gun; pin the active account explicitly.
2. **Never write a process contract you can't mechanically verify.** Every "MUST" needs a shell-scriptable check. The old contract was prose-only and subagents compiled convincing-looking reports while doing the wrong thing.
3. **Verify subagent claims against the API, not the prose.** `gh pr view` is the source of truth. Subagent text summaries are hints, not evidence.
4. **Don't confuse committer, author, and PR author.** They are three different identities with three different effects. CodeRabbit entitlement = PR author (GitHub login). Commit visibility = commit author email. Audit trail = committer.
5. **Check tool-tier assumptions with data, not assumption.** We nearly believed a "Free plan" diagnosis that was actually just "wrong user on a Pro account". Only the PR #9 vs #10/#11 comparison revealed the real pattern.

## v2 addendum — discoveries from 2026-04-24 dp-process-01 research session

While hardening the contract into v2 (see `.sisyphus/plans/dp-process-01-ralph-loop-v2.md`), live research against `docs.coderabbit.ai` and live API testing revealed four additional issues in the v1 contract that go beyond the original PR-author incident:

### Additional root cause: auto-review disabled on non-default base branches

CodeRabbit's documented default (`auto_review.base_branches: []`) means it only auto-reviews PRs targeting the **repository's default branch** (master/main). Our dp-NN PRs always target `dynamic-slider/1.4.7` (ObsidianNetwork) or `dynamic-slider` (dynamic-pterodactyl) — neither is the default branch. This means:
- Every push to every dp-NN PR required a manual `@coderabbitai review` mention to trigger review.
- The v1 contract taught the agent to wait 20 min then post the mention — correct behaviour but a symptom workaround, not a fix.

**Fix shipped**: `.coderabbit.yaml` with `auto_review.base_branches: ["dynamic-slider.*"]` committed to `master` of ObsidianNetwork (PR #5) and to `dynamic-slider` of dynamic-pterodactyl (PR #13). Auto-review now fires on integration-branch PRs without any manual trigger.

### Additional root cause: auto_pause_after_reviewed_commits: 5 (default)

CR's default pauses auto-review after 5 reviewed commits since the last pause. On longer boulders with many iterative commits, CR would silently stop reviewing after 5 pushes. We likely hit this on some dp-NN sessions and attributed the silence to slowness.

**Fix shipped**: `.coderabbit.yaml` sets `auto_pause_after_reviewed_commits: 0` (never pause).

### Observed incident: mention spam + typo handle (2026-04-24, dp-core-02 PR #4)

During the dp-core-02 ralph-loop session, the subagent posted 5 `@coderabbitai` mentions in 15 minutes:
- 12:25:15 `@coderabbitai review`
- 12:25:22 `@coderabbit resume` (typo handle — no `ai` suffix; CR does not respond to `@coderabbit`)
- 12:34:35 `@coderabbit review` (typo handle again)
- 12:34:42 `@coderabbitai review`
- 12:35:37 `@coderabbitai review`
- 12:40:03 `@coderabbitai review`

Each fresh mention resets CR's internal state machine. Multiple mentions burn rate-limit budget (Pro = 5 PR reviews/hr per developer). The subagent had no cooldown and no post-mention ack verification.

**Fix shipped**: v2 contract hard rule — 120s cooldown between mentions; post-mention ack check (CR must reply within 60s); `@coderabbitai review` is now the exception not the default path (auto-review handles pushes).

### Silence-is-clean was not documented CR behavior

The v1 `ralph-loop-verify.sh` had a rule: if a `@coderabbitai review` mention aged ≥5 min with no new review, treat as clean verdict. The CR documentation has no such protocol — 'no response = approved' is our invention. It was superstition dressed as a contract.

**Fix shipped**: silence-is-clean rule removed entirely from verify.sh. Replaced by: `CodeRabbit` commit status check = `pass` (CR's actual documented signal). Phase 2 validation confirmed the status check name is literally `CodeRabbit` and transitions pending→pass on review completion.

### Rate limit correction

Contract v1 cited 6 PR reviews/hr. The docs say **5 reviews/hr on Pro** (refilling bucket, 1 review every 12 min). Chat is 50/hr. Pro+ is 10/hr. Contract v2 has the correct value.

## Follow-up: dp-process-02 (2026-04-26) — outage failure mode surfaced

### CR outage produces stuck `pending` — not a surfaced failure

Research into zeroclaw-labs/zeroclaw#1792 (Feb 2026) revealed a concrete outage scenario: CR was broken for a period, PRs auto-merged because the status check never transitioned from `pending` to `fail`. The v2 contract's verify.sh would fail generically with 'CodeRabbit status=pending' but would not distinguish review-in-progress (~14 min for large PRs, as observed on PR #7 of this session) from a real CR outage.

**Fix shipped in dp-process-02 Phase C**: verify.sh now tracks how long CR has been `pending`. When the check has been pending for ≥ 15 min, the script escalates with the status-page URL (`https://status.coderabbit.ai/`) and the outage bypass syntax instead of a generic failure message.

**Outage bypass rule added to contract**: `--allow-actionable --reason 'CR outage <date> per https://status.coderabbit.ai/<incident-id>'` is permitted ONLY when (1) CR's status has been `pending` ≥ 15 min AND (2) status.coderabbit.ai shows an active incident. Bypass usage is written to `.sisyphus/notepads/ralph-loop-waivers.jsonl`.

**Reference**: `.sisyphus/templates/ralph-loop-contract.md` §Hard rules — Outage bypass paragraph.
