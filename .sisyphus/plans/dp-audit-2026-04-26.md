## Audit categories

Each finding must be tagged with one or more of these:

### S — Security
- AuthN/AuthZ holes (missing role checks, broken policy bindings, IDOR, privilege escalation)
- Input validation gaps (SQL injection, mass assignment, unsanitised user input reaching DB / external API / shell / view)
- Sensitive data handling (logs leaking tokens, error responses leaking internals, missing rate limits on credential paths)
- CSRF / session handling issues
- Secrets in repo / config (env defaults, hardcoded credentials)
- Dependency CVEs (composer.lock check)

### F — Functional (correctness)
- Spec drift: docs claim X, code does Y (compare each spec file 01-09 against actual code)
- Race conditions (concurrent reservation creation, double-spend windows)
- Incorrect error handling (swallowed exceptions, wrong HTTP status, missing rollback on failure)
- Missing edge-case handling (empty inputs, large inputs, unicode, timezone, null relationships)
- Off-by-one / boundary errors in pricing / capacity math
- Lifecycle bugs (state transitions that shouldn't be allowed)

### O — Operational
- Observability gaps (mutations without audit-log entries, errors logged at wrong level, missing context in logs)
- Alerting gaps (failure modes that won't notify anyone)
- Configuration brittleness (no validation at boot, silent fallback to bad defaults)
- Backfill / migration safety (destructive migrations without rollback, untested down() paths)
- Cron / scheduled-task fragility (silent failures, no health check)

### P — Performance
- N+1 queries (especially in admin pages and event listeners)
- Missing indexes on hot lookup columns
- Unbounded loops over external API calls (Pterodactyl rate budget is 240/min — enforce locally)
- Synchronous work that should be queued (heavy event handlers blocking checkout)
- Cache thrash (extension is `no-cache` for Pterodactyl per DECISIONS.md, but inner queries shouldn't suffer)

### T — Tech-debt / maintainability
- Deprecated APIs still in use (Filament v3 leftovers, retired Paymenter core methods)
- Code duplication that survived dp-09's pricing-cleanup pass
- Test gaps (public methods without coverage; complex code paths without assertions)
- Brittle tests (sleep-based, time-dependent, order-dependent, network-dependent)
- Dead code (unreachable branches, unused private methods, retired feature scaffolding)
- Complexity hotspots (long methods, deep nesting, large classes by LOC)
- Stale type hints / missing return types (PHP 8.3 features not adopted)
- Spec-doc drift in 01-DATABASE.md and 04-EVENTS.md (still describe `ptero_pricing_configs` as current — known, deferred from dp-01 closeout)

## Anti-patterns to avoid

- Do NOT propose new features. Audit means "find what's broken / risky / drifted in what exists."
- Do NOT propose architectural rewrites. If something is fundamentally wrong, flag it as a finding and let the orchestrator decide whether to plan a refactor.
- Do NOT count "could-be-better" as a finding. Findings need a concrete impact.
- Do NOT count "missing tests for trivial getters" as findings. Coverage gaps must be on code that has logic.
- Do NOT pad. 5 well-substantiated findings beat 30 flimsy ones. Be ruthless.
