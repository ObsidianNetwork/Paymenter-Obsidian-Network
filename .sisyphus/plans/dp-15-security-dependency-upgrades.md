# dp-15 — Security Dependency Upgrades (CVE remediation)

**Source**: dp-audit-2026-04-26 finding F2.
**Scope**: `/var/www/paymenter/` (outer Paymenter fork, branch `dynamic-slider/1.4.7`). NOT the extension repo — composer.lock is at the outer level.
**Type**: Security patches via `composer update` for advisories flagged by `composer audit`.
**Effort**: M (half day; could be longer if framework majors require migration work).
**Severity**: high.
**Suggested branch**: `dp-15-security-deps` off `dynamic-slider/1.4.7`.

---

## Problem

`composer audit` (run from the outer repo) reports **11 advisories affecting 9 packages**, including:

- **filament/filament CVE-2025-67507 (high)**: MFA recovery codes can be used multiple times. Affected `>=4.0.0,<4.3.1`.
- **filament/tables CVE-2026-33080 (high)**: unvalidated Range/Values summarizer values can be used for XSS. Affected `>=5.0.0,<5.3.5 | >=4.0.0,<4.8.5`.
- **phpseclib/phpseclib CVE-2026-32935 (high)**: details TBD on first read of advisory.
- **directorytree/imapengine CVE-2026-2469 (medium)**: command injection via ID command parameters. Affected `<1.22.3`.
- 7 more (medium/low) — full list at advisory output time.

Even if our extension code is correct, vulnerable framework components expose the runtime surface. Filament is in-process for the entire admin UI; phpseclib is used for SSH/TLS internals; imapengine is for inbound mail (likely unused but still loaded).

## Goal

All `composer audit` advisories resolved or explicitly waived with a documented rationale. CI's `composer audit` exit code = 0 (or filtered to only show waived advisories).


## Design

### Triage approach (in this exact order)

1. **Patch-level bumps first**: try `composer update --with-all-dependencies` and see what naturally moves to safe versions. Many advisories will resolve in one shot if pinned ranges allow.
2. **Targeted minor bumps**: for any advisory not resolved by step 1, identify the minimum minor-version bump that exits the affected range. Bump that one package.
3. **Major bumps last**: any package needing a major-version bump (e.g. filament 4 → 5) needs separate analysis. Document as a separate sub-plan if it touches >5 files.
4. **Unfixed/unfixable advisories**: any advisory without a patched version yet \u2014 document in `FORK-NOTES.md` with the advisory URL, severity, and the reason we accept the risk. Add a `composer audit --ignore` config if the audit tool supports it; otherwise a comment in CI explaining the noise.

### Filament-specific gotcha

Filament 4.x → 4.3.1+ should be safe (patch range). Filament 4.x → 5.x is a major bump and would touch the entire admin UI in both the outer fork (`app/Admin/`) and the extension repo (`extensions/Others/DynamicPterodactyl/Admin/`). Stay on 4.x line if possible.

### Test budget

After dependency bumps, run the full test suite:

```bash
vendor/bin/phpunit          # outer repo
cd extensions/Others/DynamicPterodactyl && vendor/bin/phpunit   # extension
```

Both must be green before commit. Watch for:
- Filament UI test breakage (deprecated form-builder methods).
- phpseclib API drift (ssh2/key handling).
- Laravel framework moves if minor bumps cascade.

## Edits

- `composer.json` — minimum-version bumps where needed.
- `composer.lock` — autogenerated.
- Possibly `FORK-NOTES.md` — document any waived advisories.

NO code edits unless a dependency bump breaks a call site. If code changes are needed, scope-creep guard: prefer pinning to a slightly older safe version over rewriting code.

```bash
composer outdated --direct                                  # no surprise majors held back
```

CR auto-review on the PR will likely flag any breaking-change risks; address per the critical-evaluation rule.

## Commit

Single commit if it stays clean: `chore(security): bump deps for 11 composer-audit advisories`.

If a major bump is unavoidable (e.g. filament 5), split into TWO commits:
1. `chore(security): bump patch/minor deps for N advisories`
2. `chore(security): bump filament 4 -> 5 (CVE-XXX); migrate admin UI` — separate review pass.
